# whip/whep の find_header_value のヘッダ解釈を堅牢化する

- Priority: Medium
- Created: 2026-06-05
- Model: Opus 4.8
- Polished: 2026-06-06

## 目的

WHIP / WHEP の `find_header_value`（C 版）は HTTP ヘッダから指定した名前のヘッダ値を抽出する
関数だが、以下の堅牢性の問題がある: (1) 折り返しヘッダ（obs-fold による継続行）を連結せず、
継続行を独立した行として扱う、(2) 同名ヘッダが複数行ある場合に最初の 1 行しか返さない、
(3) 行末が `\n` のみ（`\r` なし）の場合は途中行の検出に失敗する。
これらを整理し、CRLF 区切りのヘッダを正しく扱えるようにして抽出を堅牢化する。

なお C++ 版（`whip.cpp` / `whep.cpp`）は `absl::StrSplit` + `regex_match` + `std::map` による
独自実装のため、本 issue の対象外とする。

## 優先度根拠

Medium。現状の実装でも一般的な `\r\n` 区切りの単純なヘッダは抽出できている。
obs-fold は RFC 7230 Section 3.2.3 で deprecated であり、実運用で遭遇する確率は低い。
ICE サーバ情報を運ぶ `Link` ヘッダの抽出に使われるが、即時にクラッシュする類のバグではない
ため High ではなく Medium とする。

## 現状

`whip.c:745-776` の `whip_find_header_value` と `whep.c:719-750` の
`whep_find_header_value` は完全に同一実装。

行末探索に `strstr(p, "\r\n")` を使用し `p = line_end + 2` で次行へ進んでいる。
行区切りは CRLF を前提としているが、以下の脆弱性がある:
- 折り返しヘッダ（継続行）を連結しない
- 最初に一致した行で即 `return` し、同名ヘッダ複数行に対応しない
- `\n` のみの行末が途中行にある場合、`strstr(p, "\r\n")` が NULL を返し全末尾を最終行と誤認する

## 設計方針

- 行区切りは CRLF（`\r\n`）として扱う。`\n` のみの行末も検出できるよう、
  `strstr(p, "\r\n")` が失敗した場合に `strchr(p, '\n')` も試行する
- 折り返しヘッダ（継続行が空白 / タブで始まる行）を、直前のヘッダ値に連結して
  1 つの値として扱う（RFC 7230 obs-fold。deprecated だが安全のため対応する）
- フィールド名と `:` の間に空白を許容しない（HTTP の規定どおり、現状のまま）
- ヘッダ名の比較は大文字小文字を無視する（現状どおり `strncasecmp`）
- `whip.c` / `whep.c` の両方に同じ修正を入れる

## 完了条件

- CRLF 区切りのヘッダから、指定した名前のヘッダ値を大文字小文字を無視して正しく抽出できる
- 折り返しヘッダ（継続行）を連結した正しい値として抽出できる
- `\n` のみの行末を含むヘッダからもヘッダ値を抽出できる
- `whip.c` と `whep.c` の双方で同じヘッダ解釈が行われる
