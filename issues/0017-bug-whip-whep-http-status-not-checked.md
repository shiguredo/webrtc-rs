# whip/whep で HTTP ステータスコードを検証する

- Priority: High
- Created: 2026-06-05
- Model: Opus 4.8
- Polished: 2026-06-06

## 目的

WHIP / WHEP のシグナリングで、サーバから返ってきた HTTP レスポンスのステータスコードを
一切検証していない。WHIP 仕様では POST に対して `201 Created` を期待するが、
現在の実装は `4xx` / `5xx` などの失敗レスポンスであってもボディを SDP として処理し続ける。
これによりエラーボディを SDP や Link ヘッダとして誤解釈し、誤動作や無言の失敗を招く。
レスポンスのステータスコードを検証し、期待した成功ステータス以外はエラーとして扱う。

## 優先度根拠

High。サーバ側が認証エラー・レート制限・内部エラーなどで非成功ステータスを返した場合に、
エラーを検知できず誤った SDP 解析へ進む。失敗が無言のまま進行して原因究明が困難になる。
シグナリングの正しさに直結するため優先度を高くする。

## 現状

全 4 ファイルのレスポンス処理が、`\r\n\r\n` によるヘッダ終端検出のみで
ステータスコードをパースしていない。

- `whip.c:1100-1134` — `strstr(resp, "\r\n\r\n")` でヘッダ終端を検出後、即座に
  Link ヘッダ抽出と SDP 処理へ進む
- `whep.c:1074-1109` — 同一構造。`whep_OnSendRequestResponse` でヘッダ終端検出後、
  すぐにボディ解析へ
- `whip.cpp:772-778` — `resp->find("\r\n\r\n")` で分割後、`absl::StrSplit` で
  ヘッダ行パースへ進むがステータスラインは無視
- `whep.cpp:512-517` — 同上

いずれもステータスラインを解釈するコードは存在しない。

## 設計方針

- レスポンスの先頭にあるステータスライン（`HTTP/1.1 <status-code> <reason>`）を
  最初の `\r\n` まで抽出し、3 桁の数値コードをパースする
- ステータスラインが `HTTP/` で始まらない場合や、数値コードがパースできない場合は
  不正レスポンスとしてエラー扱いする

期待する成功ステータスコード:

| プロトコル | メソッド | 期待ステータス |
|-----------|---------|-------------|
| WHIP | POST | `201 Created` |
| WHEP | POST | `200 OK` |

これらの範囲（`2xx`）以外はエラーとして扱い、ボディを SDP として解釈しない。

- ログメッセージは英語で記述する
- C 版・C++ 版の 4 ファイルすべてに対応する

## 完了条件

- 非成功ステータス（`4xx` / `5xx`）のレスポンスを受け取った場合、SDP 解析へ進まず
  エラーを返し、状態が `CLOSED` に遷移する
- ステータスラインが不正なレスポンスもエラーとして扱われる
- WHIP: `201 Created`、WHEP: `200 OK` の場合のみボディ処理に進む
- C 版・C++ 版の 4 ファイルで同じ検証が行われる
